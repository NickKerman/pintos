#### ALGORITHMS

>A2: Briefly describe what happens in a call to timer_sleep(),
>including the effects of the timer interrupt handler.

`timer_sleep(ticks)` returns immediately for non-positive ticks. Otherwise it disables interrupts, computes `cur->wakeup_tick = timer_ticks() + ticks`, inserts the current thread into `sleep_list` in sorted order, and blocks the thread with `thread_block()`.  
On each timer interrupt, the handler increments the global tick count, then it calls `wake_sleepers()`, which repeatedly checks the top of `sleep_list` and unblocks every thread where `wakeup_tick <= ticks`. If a newly woken thread has higher priority than the current thread, the interrupt handler requests a yield on return.


>A3: What steps are taken to minimize the amount of time spent in
>the timer interrupt handler?

The interrupt handler only looks at the top of `sleep_list`. Because the list is kept sorted by wake-up time, it can stop as soon as the first element has `wakeup_tick > ticks` (no full scan). It performs only constant-time list operations per woken thread.


#### SYNCHRONIZATION

>A4: How are race conditions avoided when multiple threads call
>timer_sleep() simultaneously?

`timer_sleep()` disables interrupts while updating shared state (`sleep_list`, `wakeup_tick`) and before blocking. This makes insertion and state changes atomic with respect to other sleepers and the timer interrupt.


>A5: How are race conditions avoided when a timer interrupt occurs
>during a call to timer_sleep()?

The critical region in `timer_sleep()` runs with interrupts disabled, so the timer interrupt cannot interleave with inserting into `sleep_list` or with blocking the thread. The interrupt handler itself runs with interrupts disabled by the interrupt framework.


#### RATIONALE

>A6: Why did you choose this design?  In what ways is it superior to
>another design you considered?

Keeping a single global list sorted by wake-up tick makes wakeups efficient (only the earliest deadlines are checked each interrupt) and eliminates busy-waiting. Compared to scanning an unsorted list each tick, this reduces interrupt time and avoids (O(n)) work when most sleepers are not ready to wake.

>B2: Explain the data structure used to track priority donation.
>Use ASCII art to diagram a nested donation.  (Alternately, submit a
>.png file.)

Each thread tracks:

`base_priority`: the priority set by `thread_set_priority`.
`priority`: the effective (possibly donated) priority.
`locks_held`: locks the thread currently holds.
`wait_lock`: the lock the thread is currently blocked on (if any).

Donation is derived from waiters on locks the thread holds: the effective priority is `max(base_priority, max(waiter priorities over all held locks))`, and this can nest by recursively considering each waiter’s effective priority (bounded to a depth of 8).

Nested donation example:

T3 (pri 50) waits on L2 held by T2
T2 (pri 20) waits on L1 held by T1
T1 (pri 10) holds L1

T3 ->donates-> T2 ->donates-> T1

Effective priorities:
T3: 50
T2: 50  (due to T3)
T1: 50  (due to T2)


#### ALGORITHMS

>B3: How do you ensure that the highest priority thread waiting for
>a lock, semaphore, or condition variable wakes up first?

All wait queues are kept in priority order using `list_insert_ordered` with a comparator that sorts by effective priority (descending), then FIFO `sched_ord`. Additionally, `sema_up`, `lock_release`, and `cond_signal` sort their wait lists right before popping to handle priority changes due to donation while blocked.


>B4: Describe the sequence of events when a call to lock_acquire()
>causes a priority donation.  How is nested donation handled?

When a thread attempts `lock_acquire()` and the lock is held, it sets `wait_lock` to that lock and joins the lock’s waiter list. It then recomputes the holder’s effective priority (which considers all waiters across all locks held) and propagates the update up the chain using `wait_lock->holder` pointers (depth-limited). Nested donation occurs naturally because a waiter’s effective priority is computed recursively.


>B5: Describe the sequence of events when lock_release() is called
>on a lock that a higher-priority thread is waiting for.

On `lock_release`, the releasing thread removes the lock from its `locks_held`, clears `lock->holder`, and recomputes its effective priority from remaining donations. It then wakes the highest-priority waiter from the lock’s waiters (after sorting) and finally yields if a higher-priority ready thread exists.


#### SYNCHRONIZATION

>B6: Describe a potential race in thread_set_priority() and explain
>how your implementation avoids it.  Can you use a lock to avoid
>this race?

`thread_set_priority()` races with concurrent donation updates (e.g., an interrupt or another thread unblocking a higher-priority waiter) if priority and ready-queue position are updated non-atomically. The implementation disables interrupts while updating `base_priority` and recomputing effective priority to keep the thread state consistent. A lock is not appropriate because interrupts and scheduler code cannot safely sleep to acquire locks; disabling interrupts is the standard approach for these scheduler-critical updates.


#### RATIONALE

>B7: Why did you choose this design?  In what ways is it superior to
>another design you considered?

Maintaining ordered queues makes selecting the next runnable/woken thread \(O(1)\) after insertion, and ensures deterministic FIFO among equal priorities. Donation is computed from lock waiters and held locks without maintaining a separate global donor list, simplifying correctness for “multiple donors” and “nested donation,” and sorting on wake avoids stale waiter ordering when a blocked thread’s effective priority changes.
